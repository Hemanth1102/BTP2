# Open Elective Recommendation System

A Neural Collaborative Filtering (NeuMF) based recommendation system that helps students select Open Elective (OE) subjects. The system ranks all eligible OEs for a student based on their academic profile, professor quality, and sentiment analysis of course feedback. An agentic AI loop retrains and improves the model at the end of each semester.

## Project Overview

In our college, Open Electives start from Semester 5. Each semester has a pool of approximately 40 OEs offered by 5 departments. Students cannot take OEs offered by their own branch. Each student picks exactly 1 OE per semester from Semesters 5, 6, and 7.

The system:
1. Ranks all 32 eligible OEs (40 pool minus 8 own branch) for each student
2. Student submits this ranked list as their preference order form
3. College allocation system uses the ranked list to assign OEs based on seat availability
4. At the end of each semester, the agent loop retrains the model on new grade and feedback data

## Model

### NeuMF (Neural Matrix Factorization)

NeuMF is the strongest model within the Neural Collaborative Filtering (NCF) framework (He et al., 2017). It combines two parallel paths:

**GMF path** (linear interactions):
```
student_embedding ⊙ oe_embedding  →  element-wise product
```

**MLP path** (non-linear interactions):
```
[student_embedding || oe_embedding]  →  FC(256) → FC(128) → FC(64)
```

**Combined output:**
```
[GMF output || MLP output]  →  FC(1) → Sigmoid  →  relevance score (0–1)
```

### Feature Vectors

**Student vector (11 dimensions):**
```
branch_CSE, branch_ECE, branch_ME, branch_CE, branch_EEE  (OHE)
cgpa
avg_core_grade
sem1_avg, sem2_avg, sem3_avg, sem4_avg
```

**OE vector (14 dimensions):**
```
branch_CSE, branch_ECE, branch_ME, branch_CE, branch_EEE  (OHE)
available_semester
oe_avg_teaching_clarity
oe_avg_course_organization
oe_avg_overall_rating
oe_avg_interaction
oe_avg_assignment_usefulness
sentiment_score
is_new_oe
is_new_prof
```

### Training Details

| Parameter | Value |
|-----------|-------|
| Embedding dim | 64 |
| MLP layers | 256 → 128 → 64 |
| Dropout | 0.3 |
| Optimizer | Adam (lr=0.001) |
| Loss | Weighted BCE |
| Negative sampling | 1:4 ratio from eligible pool |
| Early stopping | Patience = 5 epochs |
| Total parameters | 76,033 |

### Grade to Score Mapping

| Grade | Score | Label |
|-------|-------|-------|
| O | 1.0 | 1 |
| A+ | 0.9 | 1 |
| A | 0.8 | 1 |
| B+ | 0.7 | 1 |
| B | 0.6 | 1 |
| C | 0.5 | 0 |
| F | 0.0 | 0 |

Score is used as sample weight during training. Label is the BCE loss target.

### Train / Val / Test Split

```
Semester 5 interactions → train
Semester 6 interactions → validation
Semester 7 interactions → test
```

Never shuffle across semesters. This simulates real deployment — model trained on past, evaluated on future.

## Cold Start Handling

Three cold start scenarios are handled:

### 1. New Student (Semester 5 first-timers)
Student has no OE history yet.
- Find top-10 most similar students using cosine similarity on branch + CGPA + core grade profile
- Recommend OEs that similar students chose and scored B+ or above

### 2. New OE (Newly introduced subject)
OE has no interaction history.
- Find top-5 most similar existing OEs using cosine similarity on OE feature vector
- Inherit their interaction patterns as a warm start

### 3. New Professor (No feedback history)
Three-level fallback chain:
```
Level 1 → professor's own weighted reputation score
Level 2 → department average (same offering branch)
Level 3 → global average (all professors)
```
`is_new_prof = 1` flags imputed values so the model weights them lower.

### 4. Professor Change
Handled automatically — the system always uses the professor's reputation score averaged across all courses they have ever taught, so a professor teaching a new OE still has a meaningful score from their core course history.

## Sentiment Analysis

Model: `distilbert-base-uncased-finetuned-sst-2-english`

**Why DistilBERT over simpler approaches:**
- VADER and TextBlob are lexicon-based and miss context ("not boring at all" would score negative)
- DistilBERT reads the full sentence as a unit
- 60% faster than full BERT, 97% of its accuracy, fits within 4GB VRAM

**Pipeline:**
```
course_comments.csv
       ↓ preprocess.py     → lowercase, remove special chars, drop short comments
cleaned_comments.csv
       ↓ sentiment_model.py → DistilBERT inference, batch size 32
scored_comments.csv
       ↓ aggregate_sentiment.py → weighted avg per prof_id, fallback chain
prof_features.csv
```

**Score conversion:**
```
POSITIVE confidence p  →  sentiment_score = p
NEGATIVE confidence p  →  sentiment_score = 1 - p
```

## Agentic AI Loop

The agent runs at the end of each semester when new grades and feedback are available.

### Tools

| Tool | Trigger | Action |
|------|---------|--------|
| `refresh_sentiment` | New comments added | Reruns DistilBERT pipeline, updates prof_features |
| `cold_start_handler` | New students or OEs detected | Routes to content-based fallback |
| `retrain_model` | Always (end of semester) | Retrains NeuMF, rollback if NDCG degrades |
| `shap_eval` | After retrain | SHAP feature importance, checks sentiment rank |

### Decision Logic

```
New comments detected?   → refresh_sentiment  (must run before retrain)
New students detected?   → cold_start_handler per student
New OEs detected?        → cold_start_handler per OE
Always                   → retrain_model
After retrain            → shap_eval
```

### Rollback Logic

```
After retrain:
  new_model NDCG >= old_model NDCG  →  replace checkpoint
  new_model NDCG <  old_model NDCG  →  keep old checkpoint
```

Checkpoints saved as `checkpoints/model_sem{N}.pt`.

### Memory

Every semester's actions and outcomes are logged to `results/agent_memory.csv`:
```
timestamp, semester, ndcg_before, ndcg_after, ndcg_delta, replaced, sentiment_rank, observation
```

### Important — Full 32-OE Ranking

Evaluation ranks **all 32 eligible OEs** per student — not a small sample. For each test student, the model ranks their entire eligible pool and the actual choice is checked against that full ranking. This is the honest evaluation.

The negative samples in the interaction matrix exist only for training. They are never used as evaluation candidates.

```

  1. Get all 32 eligible OEs from oe_info directly
  2. Model scores all 32
  3. Sort descending → full ranked list
  4. Reveal actual OE choice
  5. Measure where it landed in the ranking
```

## Explainability

The `/explain` endpoint uses SHAP (SHapley Additive exPlanations) to explain why a specific OE was ranked where it was for a specific student.

### How it works

```
Student + OE feature vectors combined → [student_vec || oe_vec]
         ↓
SHAP KernelExplainer runs 100 perturbations
         ↓
Returns contribution of each of 25 features to the predicted score
         ↓
Separated into positive contributors (pushing score UP)
and negative contributors (pulling score DOWN)
         ↓
Converted to human-readable explanations
```

### Example output

```json
{
  "predicted_score": 0.221,
  "top_reasons": [
    "Your CGPA moderately positively influences this recommendation (+0.0285)",
    "Semester availability slightly positively influences this recommendation (+0.0162)",
    "OE is offered by ME branch slightly positively influences this recommendation (+0.0149)"
  ],
  "bottom_reasons": [
    "No features are negatively influencing this recommendation"
  ]
}
```

### Note on performance

SHAP KernelExplainer runs 100 perturbations per call — expect 5–10 seconds per request. This is acceptable for a demo but would need optimization (pre-computed SHAP values or faster explainer) for production at scale.

## Key Design Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
